# Голосовой канал Salute (SberBoom)

## Что это

Интеграция souz с голосовым навыком Sber "Салют" — позволяет обращаться к агенту
souz голосом через умную колонку (SberBoom/StarOS) так же, как через Telegram или
десктоп, только с озвучкой ответа и световой индикацией "агент думает" вместо
текста. В коде канал называется **`salute`** (backend-пакет `ru.souz.backend.salute`,
публичные пути `/salute/webhook` и `/salute/ws`).

Отличие от привычной для Sber-платформы схемы: голосовой ассистент колонки
(распознавание речи, wake word, сама колонка) остаётся штатным — Sber сама
распознаёт речь и присылает уже готовый текст. А вот **весь интеллект (LLM-агент)
работает на backend союза**, а не на самой колонке. Колонка — тонкий исполнитель:
получает по сети команду "озвучь этот текст" или "мигни светодиодом" и просто её
выполняет.

## Из каких частей состоит

```
Sber "Салют"                 souz backend                    Колонка
(ASR/wake word,   ──HTTP──>  /salute/webhook   ──WS──>   souz-thin-client
 сама платформа)              /salute/ws       <──WS──   (тонкий клиент)
```

1. **souz backend** (этот репозиторий, пакет `ru.souz.backend.salute`)
   — принимает вебхук от платформы Sber, ведёт привязку "устройство → пользователь
   souz", запускает ход агента через уже существующий `AgentExecutionService`
   (тот же движок, что использует Telegram-канал и десктоп), и отправляет
   результат на колонку по WebSocket.
2. **souz-thin-client** (отдельный репозиторий) — маленький C++-демон, который
   постоянно работает на самой колонке. Не содержит никакой логики агента —
   только подключение по WebSocket к backend'у и исполнение команд, которые
   backend присылает (озвучить текст, мигнуть светодиодом, в перспективе —
   выполнить произвольную shell-команду).
3. **Postgres** — таблица `salute_device_bindings`: связывает `device_id`
   (аппаратный идентификатор конкретной колонки) с пользователем souz и с
   souz-чатом, в котором ведётся переписка с этим устройством. Привязка
   создаётся автоматически при первом обращении устройства.

## Как это работает — пошагово

1. Пользователь говорит колонке команду. Платформа Sber сама распознаёт речь и
   присылает POST-запрос на `/salute/webhook` с уже готовым текстом и
   идентификатором устройства (`deviceId`).
2. Backend souz:
   - если это первое обращение этого устройства — создаёт для него souz-чат и
     запись в `salute_device_bindings` (привязка к единственному текущему
     пользователю souz);
   - **сразу** отвечает платформе Sber пустым ответом (сессия остаётся
     открытой, ничего не произносится) — это позволяет не укладываться в
     жёсткий тайм-аут ответа платформы, пока агент думает;
   - параллельно, в фоне, запускает обычный ход агента через
     `AgentExecutionService.executeChatTurn(...)` — тот же самый код, которым
     пользуется Telegram-интеграция и десктопное приложение. К тексту
     запроса добавляется короткая инструкция для модели: отвечать
     разговорно, 1-3 предложениями, без markdown-разметки — потому что ответ
     будет озвучен вслух, а не показан на экране.
3. Пока агент считает ответ, backend отправляет колонке (по уже установленному
   WebSocket-соединению) команду "включи индикатор ожидания" и озвучивает
   короткую фразу-заглушку ("Секунду...", "Момент..." и т.п., случайную из
   набора).
4. Когда агент закончил ход, backend отправляет колонке команду "выключи
   индикатор" и команду "озвучь вот этот текст".
5. Колонка (souz-thin-client) выполняет обе команды локально: озвучка идёт
   через штатный системный TTS-инструмент колонки, индикатор — через
   стандартный Linux-интерфейс управления светодиодом устройства.

## Чем backend и колонка обмениваются по WebSocket

Один и тот же протокол в обе стороны — простые JSON-сообщения с полем `type`.

**Колонка → backend**, при подключении и периодически:
- `register` — колонка представляется своим `device_id` сразу после
  установки соединения;
- `ping` / `pong` — поддержание соединения живым;
- `exec_result` — результат выполнения команды (код возврата, вывод),
  отправляется в ответ на `exec`.

**Backend → колонка**, во время хода агента:
- `exec` — единственный "рабочий" тип сообщения от backend'а к колонке:
  команда для выполнения на устройстве (что запустить, с каким таймаутом).
  И озвучка текста, и управление индикатором — это просто разные `exec`-команды,
  колонка не различает их по смыслу, только выполняет.

Важное архитектурное решение: колонка не содержит захардкоженной логики "как
озвучивать" или "как мигать светодиодом" — она умеет только "выполнить команду
и вернуть результат". Все конкретные команды (что именно запустить на
устройстве) собирает backend. Это и даёт задел на будущее: та же самая
`exec`-команда в дальнейшем сможет использоваться агентом как полноценный
инструмент — "выполнить shell-команду на моей колонке" — без изменения
протокола.

## Ограничения текущей версии

- **Без аутентификации.** Ни вебхук, ни подключение колонки по WebSocket
  сейчас не проверяются — забота о безопасности (например, привязка по
  одноразовому секрету, как это уже сделано для Telegram-бота) отложена как
  осознанный технический долг. Пока `exec`-команды формирует исключительно
  backend (не пользовательский ввод и не решение агента), риск ограничен.
- **Одна колонка = один souz-чат.** Вся история переписки с конкретным
  устройством копится в одном и том же чате, привязанном к этому `device_id`.
- Состояние привязок "какая колонка сейчас на связи" хранится в памяти
  процесса backend'а (не переживает перезапуск) — колонка просто
  переподключается заново.
