# Голосовой канал Salute (SberBoom)

## Что это

Интеграция souz с голосовым навыком Sber "Салют" — позволяет обращаться к агенту
souz голосом через умную колонку (SberBoom/StarOS) так же, как через Telegram или
десктоп, только с озвучкой ответа и световой индикацией "агент думает" вместо
текста. В коде канал называется **`salute`** (backend-пакет `ru.souz.backend.salute`,
публичные пути `/salute/webhook` и `/salute/ws`).

Отличие от привычной для Sber-платформы схемы: голосовой ассистент колонки
(распознавание речи, wake word, сама колонка) остаётся штатным — Sber сама
распознаёт речь и присылает уже готовый текст. А вот **весь интеллект (LLM-агент)
работает на backend союза**, а не на самой колонке. Колонка — тонкий исполнитель:
получает по сети команду "озвучь этот текст" или "мигни светодиодом" и просто её
выполняет.

У канала два разных применения: (1) собственно голосовой диалог с колонкой
(этот раздел ниже) и (2) исполнение shell-команд скиллов прямо на колонке
через тул `RunSkillCommand` — колонка выступает четвёртой целью для
`RuntimeSandbox`, наравне с Local/Docker/Android (раздел
[«Выполнение скриптов скиллов на устройстве»](#выполнение-скриптов-скиллов-на-устройстве-runskillcommand)).
Оба используют один и тот же WebSocket-канал и один и тот же `exec`-примитив.

## Из каких частей состоит

```
Sber "Салют"                 souz backend                    Колонка
(ASR/wake word,   ──HTTP──>  /salute/webhook   ──WS──>   souz-thin-client
 сама платформа)              /salute/ws       <──WS──   (тонкий клиент)
```

1. **souz backend** (этот репозиторий, пакет `ru.souz.backend.salute`)
   — принимает вебхук от платформы Sber, ведёт привязку "устройство → пользователь
   souz", запускает ход агента через уже существующий `AgentExecutionService`
   (тот же движок, что использует Telegram-канал и десктоп), и отправляет
   результат на колонку по WebSocket.
2. **souz-thin-client** (отдельный репозиторий) — маленький C++-демон, который
   постоянно работает на самой колонке. Не содержит никакой логики агента —
   только подключение по WebSocket к backend'у и исполнение команд, которые
   backend присылает: озвучить текст, мигнуть светодиодом, или, с недавнего
   времени, выполнить произвольную shell-команду по запросу скилла (см.
   раздел [«Выполнение скриптов скиллов на устройстве»](#выполнение-скриптов-скиллов-на-устройстве-runskillcommand) ниже).
3. **Postgres** — таблица `salute_device_bindings`: связывает `device_id`
   (аппаратный идентификатор конкретной колонки) с пользователем souz и с
   souz-чатом, в котором ведётся переписка с этим устройством. Привязка не
   создаётся автоматически — её нужно явно завести командой `/salute_bind`
   в Telegram (см. [«Привязка устройства к
   пользователю»](#привязка-устройства-к-пользователю) ниже).

## Как это работает — пошагово

1. Пользователь говорит колонке команду. Платформа Sber сама распознаёт речь и
   присылает POST-запрос на `/salute/webhook` с уже готовым текстом и
   идентификатором устройства (`deviceId`).
2. Backend souz:
   - если для этого `device_id` ещё нет записи в `salute_device_bindings` —
     устройству озвучивается "Устройство не привязано.", ход агента не
     запускается (см. ниже, как привязать устройство);
   - **сразу** отвечает платформе Sber пустым ответом (сессия остаётся
     открытой, ничего не произносится) — это позволяет не укладываться в
     жёсткий тайм-аут ответа платформы, пока агент думает;
   - параллельно, в фоне, запускает обычный ход агента через
     `AgentExecutionService.executeChatTurn(...)` — тот же самый код, которым
     пользуется Telegram-интеграция и десктопное приложение. К тексту
     запроса добавляется короткая инструкция для модели: отвечать
     разговорно, 1-3 предложениями, без markdown-разметки — потому что ответ
     будет озвучен вслух, а не показан на экране.
3. Пока агент считает ответ, backend отправляет колонке (по уже установленному
   WebSocket-соединению) команду "включи индикатор ожидания" и озвучивает
   короткую фразу-заглушку ("Секунду...", "Момент..." и т.п., случайную из
   набора).
4. Когда агент закончил ход, backend отправляет колонке команду "выключи
   индикатор" и команду "озвучь вот этот текст".
5. Колонка (souz-thin-client) выполняет обе команды локально: озвучка идёт
   через штатный системный TTS-инструмент колонки, индикатор — через
   стандартный Linux-интерфейс управления светодиодом устройства.

## Чем backend и колонка обмениваются по WebSocket

Один и тот же протокол в обе стороны — простые JSON-сообщения с полем `type`.

**Колонка → backend**, при подключении и периодически:
- `register` — колонка представляется своим `device_id` сразу после
  установки соединения;
- `ping` / `pong` — поддержание соединения живым;
- `exec_result` — результат выполнения команды (код возврата, вывод),
  отправляется в ответ на `exec`.

**Backend → колонка**, во время хода агента:
- `exec` — единственный "рабочий" тип сообщения от backend'а к колонке:
  команда для выполнения на устройстве (что запустить, с каким таймаутом).
  И озвучка текста, и управление индикатором — это просто разные `exec`-команды,
  колонка не различает их по смыслу, только выполняет.

Важное архитектурное решение: колонка не содержит захардкоженной логики "как
озвучивать" или "как мигать светодиодом" — она умеет только "выполнить команду
и вернуть результат". Все конкретные команды (что именно запустить на
устройстве) собирает backend. Именно это позволило без изменения протокола
добавить второе применение той же самой `exec`-команды: полноценный
инструмент агента — "выполнить shell-команду на моей колонке", описанный
ниже.

## Выполнение скриптов скиллов на устройстве (RunSkillCommand)

Кроме голосового диалога, колонка теперь доступна агенту как ещё одна цель
для исполнения shell-команд из скиллов — наравне с локальным запуском и
Docker-песочницей. Технически это четвёртый режим общей абстракции
`RuntimeSandbox` (`:sharedLogic`, `ru.souz.runtime.sandbox`) —
`SandboxMode.SALUTE`. Сама реализация целиком в backend
(`backend/src/main/kotlin/ru/souz/backend/salute/sandbox/`), поскольку
только backend знает, какие устройства сейчас подключены.

### Как выбирается провайдер (сэндбокс)

Общий `ToolInvocationRuntimeSandboxResolver` — тот, что резолвит Local/Docker
для файловых тулов (`ToolListFiles`, `ToolDeleteFile` и т.п.) и для хранилища
самих скилл-бандлов — Salute не касается вообще. Если бы он стал
Salute-aware, любой файловый тул или загрузка бандла скилла для пользователя
с подключённой колонкой начали бы уезжать на устройство, а у него нет ни
файловой системы, которую можно было бы отдать наружу, ни доступа к бандлам.

Вместо этого Salute-aware резолвер (`BackendSaluteAwareToolInvocationRuntimeSandboxResolver`)
подключён **только** к одному конкретному тулу — `RunSkillCommand`. Это
decorator поверх обычного резолвера: он либо возвращает Salute-сэндбокс,
либо делегирует в обычный Local/Docker.

Логика выбора устройства на каждый вызов `RunSkillCommand`:
1. Если ход инициирован самой колонкой (голосом, через `/salute/webhook`) —
   используется именно то устройство, с которого пришёл запрос. Device id
   прокидывается через `ToolInvocationMeta.attributes["saluteDeviceId"]`,
   который `SaluteWebhookService` проставляет на каждый такой ход.
2. Если явного device id нет (вызов пришёл из Telegram или десктопа) —
   берётся единственное подключённое сейчас устройство, привязанное к этому
   пользователю:
   - нет ни одного привязанного устройства — вызов уходит в обычный
     Local/Docker, как будто Salute не существует;
   - устройство привязано, но сейчас не на связи — ошибка;
   - подключено больше одного устройства сразу — тоже ошибка (нужна
     дополнительная адресация, которой пока нет — см. ограничения ниже).
3. В обоих случаях устройство должно быть подключено по WebSocket прямо
   сейчас — резолвер не ходит в БД за привязками на каждый вызов, а смотрит
   на in-memory состояние `SaluteDeviceConnectionRegistry`, которое
   пополняется при каждом `register`.

### Привязка к пользователю для разных каналов

Каждая колонка привязана к тому пользователю souz, который её заклеймил
командой `/salute_bind` (см. ниже) — своя колонка у каждого пользователя,
никакого общего дефолтного аккаунта больше нет.

Практическое следствие: чтобы вызов `RunSkillCommand` из Telegram или
десктопа попал на колонку, разговор должен идти под тем же userId, на
который привязано устройство. Для самого голосового канала это не имеет
значения — device id и так приходит напрямую из атрибутов хода.

## Привязка устройства к пользователю

Привязки не создаются автоматически — их нужно явно завести, отправив
собственному Telegram-боту (уже привязанному через `/start`, см.
`backend/src/main/kotlin/ru/souz/backend/telegram/TelegramBotBindingService.kt`)
команду:

```
/salute_bind <deviceId>
```

`deviceId` пользователь берёт из приложения "Салют" (это вне зоны
ответственности backend'а — сам backend его нигде не показывает и не
озвучивает). Обработка команды перехватывается на уровне
`TelegramBotPollingService` — так же, как `/start <secret>` — и не доходит
до LLM-агента, чтобы биндинг нельзя было спровоцировать через prompt
injection из скилла или чужого сообщения.

Правила `SaluteDeviceBindingService.bind(userId, deviceId)`:
- устройство ещё никем не занято → создаётся souz-чат и привязка на
  вызвавшего пользователя;
- устройство уже привязано к тому же пользователю → ответ "уже привязано",
  без побочных эффектов;
- устройство уже привязано к **другому** пользователю → запрос отклоняется,
  существующая привязка не трогается.

Важное ограничение: `deviceId` — не секрет (это просто аппаратный
идентификатор), в отличие от одноразового `linkSecret` у Telegram-биндинга.
Значит `/salute_bind` защищает только от угона уже занятого устройства
(см. правило выше), но не от гонки за ещё не привязанным устройством — кто
первый отправит команду с правильным `deviceId`, тот его и заберёт.
Осознанно принятый компромисс для тестового инстанса с небольшим кругом
доверенных пользователей; если аудитория вырастет, стоит заменить
`deviceId` на одноразовый PIN, который колонка озвучивает при первом
неопознанном подключении.

### Как физически исполняется команда

Протокол `exec`/`exec_result` (см. выше) до этой фичи был fire-and-forget —
backend слал команду и не ждал ответа осмысленно (только логировал). Теперь
это полноценный suspend-запрос с таймаутом: `SaluteExecRequestRegistry`
хранит `id → ожидающий вызов` до прихода `exec_result` с тем же `id`, либо
до истечения таймаута, либо до разрыва WebSocket-соединения — тогда все
ожидающие запросы для этого устройства сразу завершаются ошибкой, не
дожидаясь полного таймаута. Пришедший позже `exec_result` на уже снятый
запрос тихо игнорируется.

Так как колонка физически не имеет доступа к файловой системе backend'а, у
`RunSkillCommand` для Salute есть жёсткие ограничения по сравнению с
Local/Docker/Android:
- **Скилл не может использовать вспомогательные файлы.** Если бандл скилла
  содержит что-то кроме `SKILL.md`, вызов на Salute-сэндбоксе отклоняется до
  попытки исполнения — на устройство можно доставить только текст одного
  inline-скрипта, а не файл или директорию.
- **`scriptPath` никогда не отправляется на устройство как путь.** Если
  агент передал путь к файлу в бандле, backend сам читает его содержимое
  локально и отправляет на колонку как inline-скрипт — устройство никогда
  не получает путь, который для него всё равно ничего не значит.
- **`workingDirectory` и `environment` до устройства не долетают вообще**
  (включая `SOUZ_SKILL_ROOT`/`SOUZ_SKILL_ID`/`SOUZ_SKILL_SUPPORTING_FILES`)
  — в wire-протоколе просто нет для них полей, а расширять протокол
  означало бы менять ещё и `souz-thin-client`.
- Интерпретатор на устройстве — **`sh` (POSIX), не `bash`** — как и для
  Android-сэндбокса: на встраиваемой прошивке колонки `bash` не
  гарантированно есть.

## Ограничения текущей версии

- **Без аутентификации.** Ни вебхук, ни подключение колонки по WebSocket
  сейчас не проверяются — забота о безопасности (например, привязка по
  одноразовому секрету, как это уже сделано для Telegram-бота) отложена как
  осознанный технический долг. Изначально это было приемлемо, потому что
  `exec`-команды формировал исключительно backend. С появлением
  `RunSkillCommand` это уже не так: `exec` теперь может быть инициирован
  скиллом по решению агента. Риск сейчас ограничен тем, что скиллы проходят
  structural/static/LLM-валидацию перед активацией (см.
  `agent/skills/validation`) — но отдельной авторизации самого exec-канала
  по-прежнему нет, и это стоит закрыть до расширения аудитории пользователей
  с колонками.
- **Одна колонка = один souz-чат.** Вся история переписки с конкретным
  устройством копится в одном и том же чате, привязанном к этому `device_id`.
- **Привязка "первый пришёл — тот и забрал".** `/salute_bind` не защищён
  секретом (см. [«Привязка устройства к
  пользователю»](#привязка-устройства-к-пользователю)) — только от угона уже
  занятого устройства, не от гонки за ещё не привязанным.
  Если у одного пользователя подключено сразу несколько колонок,
  `RunSkillCommand` без явного device id (см. выше) откажет с ошибкой
  неоднозначности — способа явно выбрать нужную колонку из чата пока нет.
- Состояние привязок "какая колонка сейчас на связи" хранится в памяти
  процесса backend'а (не переживает перезапуск) — колонка просто
  переподключается заново. После деплоя backend'а колонка должна успеть
  переподключиться, прежде чем резолвер `RunSkillCommand` снова увидит её
  как принадлежащую пользователю.
- **Нет отмены уже отправленной команды.** Если ход агента, вызвавший
  `RunSkillCommand`, отменяется, пока команда выполняется на устройстве —
  устройство всё равно доведёт её до конца, backend просто перестаёт ждать
  результат.

## Где искать код

- Backend, голосовой канал: `backend/src/main/kotlin/ru/souz/backend/salute/`
  — модели, роуты, сервис вебхука, реестр WebSocket-подключений, команды для
  устройства. Миграция БД: `backend/src/main/resources/db/migration/V9__salute_device_bindings.sql`.
- Привязка устройства к пользователю: `SaluteDeviceBindingService.kt` (правила
  claim'а), перехват команды `/salute_bind` — в
  `backend/src/main/kotlin/ru/souz/backend/telegram/TelegramBotPollingService.kt`.
- Backend, исполнение скриптов скиллов: `backend/src/main/kotlin/ru/souz/backend/salute/sandbox/`
  — `SaluteRuntimeSandbox`/`SaluteRuntimeSandboxProvider`, `SaluteSandboxCommandExecutor`,
  `SaluteSandboxFileSystem`, `BackendSaluteAwareToolInvocationRuntimeSandboxResolver`,
  `SaluteConnectedDeviceResolver`. Корреляция `exec`/`exec_result`:
  `backend/src/main/kotlin/ru/souz/backend/salute/SaluteExecRequestRegistry.kt`.
  Общая абстракция сэндбокса: `sharedLogic/src/commonJvmMain/kotlin/ru/souz/runtime/sandbox/`.
- Тесты: `backend/src/test/kotlin/ru/souz/backend/salute/` (включая `sandbox/`).
- Клиент для колонки: отдельный репозиторий `souz-thin-client` (сборка,
  деплой и устройство описаны в его собственном `README.md`/`docs/BUILD.md`).
